# 通用方案撰写指南

## 一份好方案的核心要素

### 1. 客户视角，不是产品视角

- 用客户的语言（行业术语、业务痛点）
- 强调"客户的成功"，不是"我们的产品"
- 避免大段技术参数堆砌

### 2. 结构清晰，可被快速浏览

- 目录 + 章节锚点（方便在线跳转）
- 每章节首段是结论（执行摘要风格）
- 关键数据加粗或图表化

### 3. 价值可量化

- 不要说"提升效率"，要说"减少 X 人天 / 月"
- 不要说"降低成本"，要说"年节省 XX 万"
- 给出 ROI 计算逻辑

### 4. 风险诚实

- 列出 3-5 个真实风险
- 给缓解方案
- 不要假装"零风险"

### 5. 报价透明

- 分项报价（软件 / 实施 / 硬件 / 服务）
- 付款节奏
- 增项规则

## 标准章节模板

1. **项目概述**：客户背景 + 痛点 + 目标
2. **需求分析**：业务 / 技术 / 集成 / 合规
3. **解决方案**：架构 + 模块 + 关键能力
4. **实施计划**：阶段划分 + 里程碑 + 团队
5. **风险与缓解**
6. **预算与商务**

## 常见错误

- ❌ 把 PPT 转 PDF 就当方案：缺少细节
- ❌ 全是技术名词：客户看不懂
- ❌ 没有数据：无法决策
- ❌ 风险藏着掖着：客户发现后不信任
- ❌ 报价模糊：商务环节反复扯皮

## 输出建议

- 第一稿：2 小时内完成
- 内部评审：30 分钟
- 客户沟通：面对面或视频，1 小时内
- 修订：1 稿
- 定稿：发送前自检清单（数据、风险、报价、联系人）
